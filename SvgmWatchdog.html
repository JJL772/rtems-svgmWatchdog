A Watchdog Driver for Synergy VGM Boards

Till Straumann, <strauman@slac.stanford.edu>; last updated 2003/1/13

Introduction

The VGM series boards by Synergy microsystems implement a watchdog timer in hardware. The watchdog, when enabled, resets the board upon expiration. Note that early versions of VGM boards may lack this feature. The hardware watchdog is implemented using one of the four timers (#3) present in the OpenPIC interrupt controller.

The driver described on this page initializes the watchdog timer (compile-time configurable timeout) and starts a 'watchdog task' to periodically 'pet' (reset) the watchdog timer. The scheduling priority for the task is set at driver initialization and should be carefully chosen.

The driver also implements an RPC service for remotely watching and/or resetting the VGM board.

The watchdog driver runs on vxWorks or RTEMS (a test version also runs on various flavours of Unix), the remote client on Linux, Solaris or other Unices.

Starting the Driver

The watchdog driver (along with makefiles and the host-side 'client' program) is available in CVS ($CVSROOT/drivers/svgmWatchdog). After building and linking the driver (either statically or as a loadable module) into the system, the driver must be initialized/started. The generic SSRL 'system' startup script already loads the driver but does not start it, however (to avoid interference of the watchdog task with the user application).

The watchdog is started by calling

wdStart(watchdog_task_priority)

e.g. from a startup script. The watchdog may be halted (and the watchdog task terminated) using the

wdStop()

command. (CEXP NOTE: the driver is automatically stopped when unloading by a module finalizer. It is not automatically started by the module initializer, however.)

Chosing a Proper Watchdog Priority

Chosing the watchdog task priority is a tradeoff. It is desirable to set the priority as low as possible since this will catch runaway tasks. OTOH, too low a priority may result in spurious resets if higher prioritized tasks need the CPU for longer than the watchdog expiration (1.2seconds by default) - cexp module loading or IOC initialization are examples for quite lengthy operations. In any case, the watchdog priority should be lower than any real-time critical task's. Note that the watchdog task also runs an RPC service (described below).

The problem of chosing too high a priority occurs when e.g. a networking task runs away or a deadlock occurs while the watchdog task still is operational. In these cases, control over the remote CPU may be lost unless 'remote watching' as described in the following section is used.

Using "Remote Watching"

Using the watchdog driver's RPC service functionality in combination with a host-side client program 'wdclnt', permanent control over a target VGM computer can be asserted. After a connection has been established between wdclnt and the target watchdog RPC server, the server (i.e. the watchdog driver) requires the client to send periodic 'software pet RPCs' (compile-time default: 5 seconds) and enforces a hardware reset if these do not occur. Hence, any event leading to a loss of the RPC connection (target crash or deadlock but also a broken network cable or a host crash) eventually results in a target reset. Note that it is possible to disconnect from the RPC server, thus leaving the "remote watching" mode. The wdclnt program transparently connects when starting up and disconnects upon exit.

While "remote watching" is very useful for development, it is probably not desirable to use this mode during extended periods of time on production targets.

Using "Remote Reset"

A VGM target can be remotely reset (using the wdclnt program) either by suspending a connected wdclnt (for more than 5 seconds) or by letting it send a special "reset" RPC (using a command line option) who is also recognized by the server.

Using the wdclnt Program

NAME

wdclnt - SVGM watchdog client program

SYNOPSIS

wdclnt [options] target

DESCRIPTION

wdclnt establishes a connection to the watchdog RPC server running on the target. It issues periodic 'PET' RPCs while connected. wdclnt may be terminated by sending it a SIGINT signal which will terminate the connection. Sending SIGUSR1 causes wdclnt to issue a RESET request to the target.

OPTIONS

-h
print use information and exit
-v
print version information and exit
-r
send target RESET request and exit
